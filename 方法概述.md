# 一、总体方案概述
随着大语言模型的发展，基于自然语言自动生成自动化工作流逐渐成为智能系统的重要研究方向。然而现有方法在生成可执行工作流时仍面临若干关键挑战。

首先，许多研究侧重于 **动作到节点的局部映射**，缺乏对整体执行结构的建模，导致生成流程容易出现结构冲突，如循环依赖、执行路径不完整等问题。

其次，部分方法直接生成 **自由 DAG（Directed Acyclic Graph）** 结构，虽然具有较高表达能力，但由于缺乏结构约束，模型容易生成执行系统无法解析的拓扑结构，从而影响实际部署。

再次，节点参数生成往往缺乏严格的 schema 约束，自动生成参数容易出现缺失、类型不匹配或变量引用错误的问题，导致生成流程难以直接执行。

为解决上述问题，本文提出一套 **结构驱动的自动化工作流生成方法**。该方法以 **统一任务表示（UTR）** 作为中间语义表示，并采用 **结构化工作流生成（Structured Workflow Generation）** 的思想，通过 **工作流骨架规划（Workflow Skeleton Planning）** 与 **节点填充（Node Filling）** 两阶段生成策略，实现从自然语言任务描述到可执行工作流 DSL 的自动生成。

整体流程如下：

```plain
自然语言任务
      ↓
任务解析（UTR生成）
      ↓
工作流骨架规划
      ↓
节点识别与填充
      ↓
连接关系生成
      ↓
节点参数生成
      ↓
结构校验与DSL输出
```

在结构设计上，本文不直接生成任意 DAG，而是将工作流表示为 **由控制结构组成的层次化结构树****（Workflow Skeleton Tree）**。  
系统首先生成符合执行系统规范的结构骨架，再逐步填充具体任务节点与参数，从而保证生成结果既具有较高表达能力，又具备稳定的执行结构。

**工作流骨架由若干 ****基本控制结构（Control Structures）**** 组合构成，包括：**

+ **顺序结构（Sequential）**
+ **条件分支结构（Conditional）**
+ **并行结构（Parallel）**
+ **迭代结构（Iteration）**

**通过这些结构的递归组合，可以表达复杂自动化流程，同时避免自由图生成带来的结构不稳定问题。**

**在推理机制上，本方法采用 ****大语言模型与结构规则结合的混合生成策略****：**

+ **大语言模型负责任务语义理解与节点选择**
+ **结构规则负责工作流拓扑生成与合法性约束**

**最终系统输出 ****结构合法、参数完整、可直接执行的工作流 DSL 表示****。**

# **二、详细实现步骤**
---

# **步骤1：任务解析与统一任务表示生成**
## **目标**
**将自然语言任务描述转换为结构化的 ****统一任务表示（UTR）****，为后续工作流生成提供语义基础。**

**UTR主要包含以下信息：**

+ **任务动作序列**
+ **资源对象**
+ **控制结构意图**
+ **上下文变量**

---

## **（1）任务动作解析**
**利用大语言模型解析任务中的核心操作动作。**

**例如：	**

**输入：**

```plain
下载销售数据，清洗数据，然后生成报告并发送到邮箱
```

**解析得到：**

```plain
download_data
clean_data
generate_report
send_email
```

**同时识别动作之间的时序关系。**

---

## **（2）资源对象抽取**
**从任务描述中提取任务涉及的资源对象，例如：**

| **类型** | **示例** |
| --- | --- |
| **数据** | **销售数据** |
| **文件** | **报告文件** |
| **服务** | **API** |
| **目标** | **邮箱** |


**资源结构表示为：**

```plain
Resource {
  name,
  type
}
```

---

## **（3）控制结构识别**
**识别任务中的控制逻辑，例如：**

| **描述** | **结构** |
| --- | --- |
| **如果失败则通知** | **条件** |
| **对所有文件执行** | **循环** |
| **同时处理多个任务** | **并行** |


**输出结构：**

```plain
ControlIntent {
  type,
  condition,
  target
}
```

---

## **（4）UTR结构定义**
**最终生成统一任务表示：**

```plain
UTR {
  actions,
  resources,
  control_intents,
  variables
}
```

---

# **步骤2：工作流骨架规划（Workflow Skeleton Planning）**
## **目标**
**根据UTR信息生成 ****工作流结构骨架（Workflow Skeleton）****。**

**骨架描述的是 ****控制结构组成的层次结构****，而不是具体节点。**

**骨架表示为：**

```plain
SkeletonNode {
  type,
  children
}
```

---

## **支持的控制结构**
**系统支持四类基本结构：**

### **顺序结构**
```plain
A → B → C
```

---

### **条件结构**
```plain
Condition
      ↙    ↘
     A      B
      ↘    ↙
        C
```

---

### **并行结构**
```plain
A
     ↙ ↓ ↘
    B  C  D
     ↘ ↓ ↙
       E
```

---

### **迭代结构**
```plain
Loop
  ↓
Task
  ↓
LoopEnd
```

---

**这些结构可以递归组合形成复杂流程。**

**例如：**

```plain
Sequential
  ├─ Download
  ├─ Parallel
  │    ├─ Clean
  │    ├─ Analyze
  │    └─ Summarize
  └─ SendEmail
```

---

# **步骤3：节点识别与填充**
## **目标**
**根据UTR动作，将任务节点填充到骨架结构中。**

**节点库中每个节点包含：**

```plain
node_type
input_schema
output_schema
parameter_schema
```

**节点识别流程：**

**1 动作语义向量化  
****2 向量检索候选节点  
****3 LLM语义匹配筛选**

**最终得到：**

```plain
NodeSet = {n1,n2,n3}
```

**然后按照骨架结构将节点填入。**

---

# **步骤4：连接关系生成**
## **目标**
**确定节点之间的连接关系。**

**连接生成依据：**

**1 数据依赖  
****2 资源依赖  
****3 动作顺序**

**例如：**

**如果**

```plain
nodeA.output == nodeB.input
```

**则建立连接：**

```plain
nodeA → nodeB
```

**对于控制结构：**

**并行结构生成 ****Fork / Join**** 连接。**

---

# **步骤5：节点参数生成**
**为每个节点生成参数配置。**

**参数生成依据：**

```plain
任务描述
UTR变量
节点schema
```

**例如：**

**任务描述：**

```plain
发送至邮箱 test@example.com
```

**生成：**

```plain
{
 "recipient": "test@example.com"
}
```

**参数生成必须满足 schema。**

---

# **步骤6：结构校验与DSL生成**
## **目标**
**验证生成结构并输出 DSL。**

**校验内容包括：**

| **检查项** | **方法** |
| --- | --- |
| **拓扑合法** | **DAG检测** |
| **变量引用** | **变量检查** |
| **参数完整** | **schema校验** |
| **执行路径** | **Start→End** |


**最终生成：**

```plain
WorkflowDSL {
  nodes,
  edges
}
```

**该 DSL 可直接用于执行系统运行。**
