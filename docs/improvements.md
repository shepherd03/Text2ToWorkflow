# 优化记录 (Improvements)

本文档用于记录项目开发过程中进行的核心架构优化、代码重构和健壮性提升。
如果移除了某部分优化，请同步从本文档中移除对应记录。

## 1. 骨架规划模块健壮性优化
- **优化内容**：在 `skeleton_planner.py` 中，针对 Kahn 拓扑排序算法的循环依赖问题增加了严格的异常抛出机制。
- **优化原因**：原有的兜底策略（将包含循环依赖的未入队节点强行打包到最后一层）不够严谨，未来转化为 DSL 时可能会导致执行死循环或逻辑错乱。
- **具体实现**：当检测到 UTR 的动作之间存在循环依赖时，系统将抛出 `ValueError`，并列出涉及的动作 ID，从而在骨架生成阶段直接阻断错误，防止生成非法的环状工作流骨架树。

## 2. 第三阶段规划文档补充
- **优化内容**：新增第三阶段 DSL 落地规划报告，系统化定义了从骨架树到 Dify Workflow DSL 的方法、模块拆分、控制流降级、变量绑定、兼容校验与实施路线。
- **优化原因**：第三阶段是项目从“结构生成”走向“平台可运行”的关键环节，需要一份能够直接指导后续开发的统一方案，避免实现过程中方向反复变化。
- **具体实现**：在 `docs/第三阶段DSL落地规划报告.md` 中沉淀完整规划，并明确第三阶段继续遵循“算法主导、模型辅助”的编译式落地路径，确保与现有三阶段架构及 Dify Workflow 规范兼容。

## 3. 第三阶段目标范围收敛
- **优化内容**：从第三阶段规划文档中移除了对 Chatflow 扩展的相关表述，统一收敛为仅支持 Dify Workflow。
- **优化原因**：当前项目的明确目标是生成可在 Dify Workflow 中导入并运行的 DSL，保留与 Chatflow 相关的规划会分散实现重点，也容易让后续设计边界变得模糊。
- **具体实现**：已将 `docs/第三阶段DSL落地规划报告.md` 中涉及“后续扩展 Chatflow”或“先支持 Workflow 再扩展 Chatflow”的描述改为单目标表述，确保第三阶段只围绕 Dify Workflow 兼容与落地展开。
