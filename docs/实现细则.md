# UTR 生成模块实现细则

## 1. 设计原则

- 结构优先：任何抽取结果都必须落入可校验的 UTR Schema
- 模型与规则协同：DeepSeek 负责语义泛化，规则负责稳定兜底
- 渐进可靠：即使 API Key 未配置也能完整跑通流水线

## 2. 四阶段执行细节

### 阶段一：预处理

- 标准化口语表达，减少长尾表述差异
- 将复合句切分为可抽取片段，降低多意图干扰
- 输出：
  - `cleaned_text`
  - `clauses`

### 阶段二：并行抽取

- 动作抽取：
  - 目标：产出有序动作列表
  - 兜底：动词映射 + 子句首词回退策略
- 资源抽取：
  - 目标：识别 data/file/service/target/variable
  - 兜底：URL、邮箱、路径、关键词规则
- 控制意图抽取：
  - 目标：识别 sequential/conditional/parallel/iteration
  - 兜底：触发词匹配 + 条件/循环目标片段提取
- 变量抽取：
  - 目标：抽取常量并标注类型
  - 兜底：邮箱、URL、数值、布尔、结构化文本识别

### 阶段三：整合关联

- 统一生成稳定 ID
- 动作按 `order` 重排序并去重
- 控制意图中的 `target_actions` 自动映射为动作 ID

### 阶段四：三级校验

- Schema 校验：Pydantic + JSON Schema 双校验
- 逻辑校验：
  - 动作顺序连续
  - 控制目标动作存在
  - 变量 ID 不重复
- 完整性校验：
  - 默认告警模式
  - 可切换严格模式并转为阻断错误

## 3. DeepSeek 提示词约束

统一要求模型返回 JSON 对象，禁止自然语言解释，降低解析失败概率。每个子模块只请求本模块字段，避免大对象输出带来的不稳定性。

## 4. 规则配置集中化

- 规则文件位置：`configs/rules/*.json`
- 由 `utr_generator/rulebook.py` 统一加载
- 支持通过 `UTR_RULEBOOK_PATH` 指向外部规则目录或单文件
- 预处理、动作、资源、控制意图、变量抽取均从该配置读取映射和正则
- 加载时校验必要模块，缺失时直接抛错避免静默异常
- 规则调整无需改动解析模块代码，便于论文实验中快速迭代

## 5. 稳定性策略

- LLM 调用失败自动退化到规则引擎
- JSON 解析支持代码块和噪声文本提取
- 校验报告分离 errors/warnings，便于后续人工审阅和自动化决策

## 6. 与论文后续阶段的接口

对接下游时，建议下游只消费：

- `UTR.actions`
- `UTR.control_intents`
- `UTR.variables`

并将 `validation_report` 作为质量门禁信号，低质量样本先进入人工审阅队列。
