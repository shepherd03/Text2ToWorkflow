# 第三阶段节点映射系统设计报告

## 一、报告目标

本报告聚焦第三阶段“DSL 落地”中的节点映射问题，目标是在现有三阶段架构不变的前提下，设计一套可解释、可验证、可扩展的节点映射系统。这里所说的节点映射，不是把动作名称简单翻译成某个 Dify 节点名称，而是把第二阶段骨架树中的抽象节点，稳定地定型为 Dify Workflow 可执行节点，并为后续参数补全、变量绑定、控制流翻译和平台校验提供可靠基础。

节点映射系统要解决的核心问题有三个。第一，抽象 `ActionSlot` 需要被定型为具体的平台节点，否则无法生成合法 DSL。第二，节点类型选择不能只依赖动作名称，必须结合语义、输入输出、上下文位置和平台能力综合判断。第三，映射结果必须接受可落地性校验，不能出现“语义上看起来合理，但平台上无法配置”的假成功。

## 二、设计目标与边界

### 2.1 设计目标

节点映射系统需要同时满足以下几个目标：

1. **稳定性**：同一类动作在同类上下文下应尽量得到一致的节点类型。
2. **可解释性**：每次映射都要能回答“为什么选这个节点”。
3. **可落地性**：映射结果必须能继续进入参数生成与平台校验流程。
4. **可降级性**：当高阶节点信息不足时，能够回退到保守实现，而不是中断整个流程。
5. **可扩展性**：后续支持更多 Dify 节点时，不需要重写整套决策逻辑。

### 2.2 系统边界

节点映射系统只负责回答“抽象节点应该落成什么平台节点”以及“这个决定的依据和风险是什么”，不直接负责以下事项：

- 不重新设计骨架控制流；
- 不直接渲染完整 YAML；
- 不替代变量绑定模块；
- 不替代最终平台校验器；
- 不把前两阶段改造成平台参数生成器。

换句话说，节点映射系统位于第三阶段的前半段，处在“骨架归一化”之后、“参数渲染与图构建”之前。

## 三、映射对象与输入输出

### 3.1 输入对象

节点映射系统的直接输入不应只是一个 `ActionSlot`，而应该是一个完整的“节点映射上下文”，至少包括以下信息：

1. 当前骨架节点本身；
2. 当前节点对应的 `Action` 元数据；
3. 所属控制域信息；
4. 上下游动作依赖；
5. 当前可见变量池；
6. 平台能力白名单；
7. 默认映射策略与降级策略。

建议把这部分输入在内部整理成统一上下文对象，例如：

- `action_id`
- `action_name`
- `description`
- `inputs`
- `outputs`
- `parent_block_type`
- `is_in_iteration`
- `is_in_conditional`
- `upstream_actions`
- `downstream_actions`
- `available_variables`
- `available_resources`

只有把节点放回上下文里看，映射才能稳定。脱离上下文单看动作名称，容易把“生成文本”和“格式整理”这类边界动作映射错位。

### 3.2 输出对象

节点映射系统的输出不应是字符串类型的节点名，而应是一个结构化的“节点定型结果”。建议至少包含以下字段：

1. `source_action_id`：来源动作 ID；
2. `chosen_node_type`：最终选定的平台节点类型；
3. `candidate_node_types`：候选节点列表；
4. `confidence`：当前决策置信度；
5. `decision_reason`：主要决策依据；
6. `required_params`：该节点类型所需关键参数；
7. `available_params`：当前可用参数来源；
8. `fallback_node_type`：若无法落地时的降级节点；
9. `needs_human_fill`：是否需要后续人工或规则补全；
10. `trace`：完整决策链路。

这个输出对象后续可以直接交给参数补全模块和图构建模块使用，而不是再次做二次猜测。

## 四、节点映射系统的总体结构

节点映射系统建议拆成六个连续层次，每一层只解决一类问题，前一层的输出作为后一层的输入。

### 4.1 平台能力注册层

这一层的任务不是处理具体动作，而是定义 Dify Workflow 当前允许使用的节点能力边界。建议为每一种平台节点维护一份能力注册信息，至少包括：

- 节点名称；
- 节点用途；
- 典型输入类型；
- 典型输出类型；
- 必要参数；
- 可选参数；
- 适用场景；
- 不适用场景；
- 默认降级目标。

例如：

- `llm`：适合生成、总结、改写、分析等语义生成任务；
- `code`：适合规则转换、数据清洗、聚合、保守兜底；
- `template-transform`：适合变量拼装和模板化文本生成；
- `http-request`：适合外部 API 调用；
- `if-else`：适合条件分支门；
- `iteration`：适合列表或文件集合遍历；
- `variable-aggregator`：适合多来源结果聚合。

这一层的作用是先建立“平台能做什么”的边界，避免映射系统自由发散到平台不支持的节点上。

### 4.2 语义画像提取层

这一层把抽象动作转成标准化语义画像。节点映射系统不应该直接使用 `action_name` 做最终判断，而应该先从动作与上下文中提取特征。

建议提取以下几类特征：

1. **意图特征**
   - 生成
   - 摘要
   - 改写
   - 分析
   - 请求
   - 解析
   - 合并
   - 聚合
   - 分类
   - 判断
   - 遍历

2. **输入特征**
   - 是否需要文本输入
   - 是否依赖结构化对象
   - 是否依赖文件
   - 是否依赖列表
   - 是否依赖外部服务

3. **输出特征**
   - 输出文本
   - 输出布尔值
   - 输出对象
   - 输出列表
   - 输出文件

4. **执行特征**
   - 是否为纯计算
   - 是否有外部副作用
   - 是否依赖上游多源输入
   - 是否处于循环体内部
   - 是否处于条件分支内部

5. **上下文特征**
   - 所属父块类型
   - 上游节点类型
   - 下游节点对输出的期待形式
   - 是否作为并行汇合前后的关键节点

最终，这一层产出的不再是“一个名字”，而是一份标准化语义画像，例如：

- `intent = generate_text`
- `input_shape = text`
- `output_shape = text`
- `requires_external_access = false`
- `is_in_iteration = false`

语义画像是整个映射系统稳定性的基础。

### 4.3 规则决策层

这一层负责做主映射决策，是节点映射系统的核心。建议采用“强规则优先，弱规则补充”的方式。

#### 4.3.1 强规则

强规则命中后可以直接决定节点类型，适用于边界明确的情况。例如：

- 动作位于 `ConditionalBlock` 的分支门位置，对应 `if-else`；
- 动作本身不是普通动作，而是循环容器语义，对应 `iteration`；
- 需要显式汇总多个分支输出，对应 `variable-aggregator`；
- 动作描述中出现明确的外部接口语义，并且存在 URL、method 或 service 资源线索，对应 `http-request`；
- 动作主要任务是模板拼接，不涉及复杂推理，对应 `template-transform`。

强规则的特点是解释性强、波动小，适合优先覆盖最稳定的场景。

#### 4.3.2 弱规则

弱规则用于提供候选集合和初始排序，不直接拍板。例如：

- `generate`、`summarize`、`rewrite`、`analyze` 倾向于 `llm`；
- `transform`、`merge`、`normalize`、`clean` 倾向于 `code`；
- `extract` 可能对应 `parameter-extractor`、`doc-extractor` 或 `code`；
- `search`、`lookup` 可能对应 `tool`、`http-request` 或 `knowledge-retrieval`。

弱规则的作用不是给出最终答案，而是缩小候选范围，降低后续模型判别复杂度。

### 4.4 受约束模型判别层

这一层只处理规则无法稳定决策的灰区场景。模型在这里的职责不是自由生成 DSL，而是在有限候选集合中做判别和补充说明。

建议约束模型输出以下结构：

- `chosen_type`
- `alternative_types`
- `confidence`
- `reason`
- `param_stub`

例如，当一个动作既可能是 `template-transform`，也可能是 `code` 时，可以让模型在给定候选集合中选一个最合适的类型，并说明原因。这样做的好处有两个：

1. 模型不再承担开放式生成责任，可控性更强；
2. 模型输出可以直接进入程序校验，而不是作为最终结果原样采纳。

如果模型判别结果与规则系统冲突，应优先由程序侧根据置信度和参数可满足性做最终仲裁。

### 4.5 可落地性检查层

这一步非常关键。节点映射系统不能只做“语义选择”，还必须做“落地可行性判断”。

可落地性检查至少要回答三个问题：

1. 当前选定节点类型的关键参数是否有来源；
2. 当前选定节点类型是否在 Workflow 模式内允许使用；
3. 当前选定节点类型生成后，后续变量绑定是否可继续推进。

例如：

- 一个动作语义上像 `http-request`，但没有可靠 URL 或服务来源，这时就不能直接认定为合法 `http-request`；
- 一个动作语义上像 `iteration`，但找不到列表或文件集合输入，也不能直接渲染为标准循环节点；
- 一个动作语义上像 `tool`，但 provider 和参数结构都缺失，这时应进入降级路径，而不是保留一个无法运行的工具节点。

这一层的作用是把“映射正确”和“可以落地”区分开。只有同时满足两者，才能进入最终节点定型。

### 4.6 降级与标记层

节点映射系统不能追求零失败，而应该追求“失败时仍然可控”。因此必须显式设计降级策略。

建议建立如下几类降级路径：

1. `tool -> http-request -> code`
2. `http-request -> code`
3. `template-transform -> code`
4. `llm -> code`
5. `parameter-extractor -> code`
6. `doc-extractor -> code`
7. `iteration -> 待补全状态或保守代码节点`

降级时需要同时输出风险标记，例如：

- `degraded = true`
- `degrade_reason = 缺少关键参数`
- `risk_level = medium/high`
- `follow_up_required = true`

这样后续报告模块、验证模块和调试模块都能明确知道这个节点不是“原生精确映射”，而是保守替代实现。

## 五、映射系统的完整决策流程

对于每个 `ActionSlot`，建议使用如下标准决策顺序：

1. 识别该节点是普通业务动作，还是由控制流派生出的特殊节点；
2. 提取动作语义画像；
3. 从平台能力注册表中召回候选节点类型；
4. 使用强规则做第一轮判定；
5. 使用弱规则做候选排序；
6. 如果存在歧义，调用受约束模型判别；
7. 对最优候选执行可落地性检查；
8. 若通过，输出最终节点定型结果；
9. 若不通过，进入降级策略；
10. 记录 trace 和风险标签。

这个流程的重点不在于“尽可能早决定”，而在于“每一步都能解释为什么继续向下，以及为什么最终选了这个节点”。

## 六、不同类型抽象节点的映射策略

### 6.1 ActionSlot 的映射策略

`ActionSlot` 是节点映射系统的主对象，建议分为以下几类语义：

1. **生成与改写类**
   - 典型关键词：generate、summarize、rewrite、analyze
   - 主映射：`llm`
   - 降级：`code`

2. **规则转换与整理类**
   - 典型关键词：transform、merge、clean、normalize、compute
   - 主映射：`code`
   - 降级：仍为 `code`

3. **模板拼接类**
   - 典型关键词：format、compose、fill_template
   - 主映射：`template-transform`
   - 降级：`code`

4. **外部调用类**
   - 典型关键词：request、fetch_api、post_data、call_service
   - 主映射：`http-request`
   - 降级：`code`

5. **工具调用类**
   - 典型关键词：call_tool、search_web、use_plugin
   - 主映射：`tool`
   - 降级：`http-request` 或 `code`

6. **提取与解析类**
   - 典型关键词：extract、parse、read_document
   - 主映射：根据对象不同在 `doc-extractor`、`parameter-extractor`、`code` 之间选择
   - 降级：`code`

### 6.2 ConditionalBlock 的映射策略

`ConditionalBlock` 不应继续作为抽象结构保留到 DSL 输出阶段，而应在控制流翻译层统一映射为 `if-else` 节点。这里需要注意两件事：

1. 分支门节点本身由 `if-else` 承接，不再把条件动作当成普通 `ActionSlot` 处理；
2. `condition_description` 需要进一步转成结构化 `cases.conditions`。

因此，条件块的映射实际上是“控制节点映射”，不是“动作节点映射”。

### 6.3 LoopBlock 的映射策略

`LoopBlock` 应统一映射为 `iteration` 容器及其内部 `iteration-start` 结构。循环体内部的业务动作仍然按普通 `ActionSlot` 映射。

也就是说，循环映射要分两层：

1. 容器层：由 `LoopBlock` 映射为 `iteration`；
2. 体内层：由循环体中的 `ActionSlot` 分别映射为业务节点。

这里最关键的不是“看见循环就生成 iteration”，而是先定位 iterator 来源和输出聚合方式。

### 6.4 ParallelBlock 的映射策略

`ParallelBlock` 不是单一平台节点，而是一种子图展开策略。它自身不直接落成某个业务节点，而是展开为：

1. 多个分支入口；
2. 分支内部节点子图；
3. 必要时的显式汇合节点。

并行块映射的重点不在于“选节点类型”，而在于“决定是否插入 `variable-aggregator` 或 barrier 型兜底节点”。

### 6.5 SequentialBlock 的映射策略

`SequentialBlock` 通常不映射为独立平台节点，而是作为子图连接策略存在。其主要职责是维护顺序关系，把前一个子图出口接到后一个子图入口。

## 七、参数可满足性判定标准

节点映射系统建议为不同节点定义最小参数门槛，只有达到门槛后，才算真正可以落地。

### 7.1 llm

最小门槛：

- 至少存在任务意图或动作描述，可生成基础 prompt；
- 至少存在一种稳定输入来源；
- 至少能确定一个输出字段。

### 7.2 http-request

最小门槛：

- 存在 URL 或可推断服务端点；
- 存在 method 或默认规则可推导；
- 输入参数来源明确。

### 7.3 template-transform

最小门槛：

- 存在模板骨架；
- 变量来源明确；
- 输出目标为文本或可模板化结果。

### 7.4 iteration

最小门槛：

- 存在可识别的数组或文件集合输入；
- 能确定循环体起始节点；
- 能确定循环后的输出形式。

### 7.5 variable-aggregator

最小门槛：

- 至少存在两个可聚合来源；
- 下游确实需要统一结果；
- 聚合输出类型可确定。

有了这套门槛后，映射系统就不再只依赖语义猜测，而是具备了明确的“放行条件”。

## 八、置信度与 trace 设计

为了让映射系统可调试、可评估，建议引入置信度与 trace 机制。

### 8.1 置信度来源

置信度可以综合以下几类信号：

1. 强规则是否命中；
2. 候选节点之间分数差距是否明显；
3. 模型判别是否一致；
4. 参数可满足性是否完整；
5. 上下文约束是否支持当前选择。

可采用简单三级置信度：

- `high`：强规则命中且参数完整；
- `medium`：弱规则 + 模型一致，参数基本齐全；
- `low`：语义不确定或需降级。

### 8.2 trace 内容

建议每次映射记录以下信息：

- 原始动作名称与描述；
- 提取出的语义特征；
- 候选节点集合；
- 规则命中情况；
- 模型判断结果；
- 参数可满足性检查结果；
- 最终节点类型；
- 是否降级；
- 风险标签。

trace 的意义在于后续如果某个节点映射错误，可以准确回溯是语义提取出了问题，还是规则排序出了问题，还是平台参数根本不够。

## 九、建议的实现模块划分

为了便于第三阶段后续真正开发，建议把节点映射系统拆成以下几个子模块：

### 9.1 `capability_registry.py`

负责维护平台节点能力白名单与每类节点的能力描述、必要参数、降级目标。

### 9.2 `semantic_profiler.py`

负责从 `Action`、上下文和变量池中提取标准化语义画像。

### 9.3 `rule_engine.py`

负责强规则与弱规则匹配，并输出候选节点排序结果。

### 9.4 `llm_mapper.py`

负责在候选集合内执行受约束判别，不直接生成最终 DSL。

### 9.5 `feasibility_checker.py`

负责检查当前候选节点的关键参数是否可满足，是否可进入实际渲染阶段。

### 9.6 `fallback_manager.py`

负责为不可落地节点选择降级目标并打标。

### 9.7 `node_mapper.py`

作为总调度器，对外暴露统一入口，串联上述模块并返回最终节点定型结果。

## 十、与现有项目架构的兼容方式

这套节点映射系统可以直接作为第三阶段内部模块引入，不需要改动第一阶段和第二阶段的核心接口。

推荐的接入位置如下：

1. 读取 `UTR` 与骨架树；
2. 做骨架归一化；
3. 对每个 `ActionSlot` 调用节点映射系统；
4. 生成带平台类型的中间节点对象；
5. 再进入控制流翻译、变量绑定、图构建和校验流程。

这样做的好处是，节点映射系统成为第三阶段内部的一个专门能力模块，而不是把平台语义前置到前两阶段。

## 十一、建议的第一批实现范围

节点映射系统不适合一开始追求全覆盖，建议优先覆盖最常见且最稳的节点类型：

1. `start`
2. `end`
3. `llm`
4. `code`
5. `if-else`
6. `iteration`
7. `template-transform`
8. `http-request`
9. `variable-aggregator`

这一批足以覆盖顺序流程、文本生成、规则转换、简单 API 调用、条件判断、列表遍历和多分支汇总等主要场景，也最适合先形成第三阶段最小闭环。

## 十二、阶段性验收标准

节点映射系统是否设计成功，不应只看“能不能输出一个节点类型”，而要看以下几项是否同时成立：

1. 对典型动作样例能够稳定输出一致的节点类型；
2. 映射结果带有清晰的理由、候选和风险标记；
3. 高阶节点在参数不足时能够正确降级；
4. 控制节点与业务节点不会混淆处理；
5. 后续变量绑定和图构建能够直接消费映射结果；
6. 失败样例能够通过 trace 回溯问题来源。

如果这几项能成立，说明节点映射系统已经从“经验判断”变成了“可复用的工程模块”。

## 十三、总结

节点映射系统的关键，不在于把每个动作一次性猜成最优平台节点，而在于建立一套稳定的决策链：先理解平台能力，再提取动作语义，再通过规则与受约束模型做候选筛选，然后用可落地性检查决定是否放行，最后用降级策略兜住不确定场景。

对第三阶段来说，这套系统的价值有三点。第一，它把节点定型从零散经验判断变成结构化流程。第二，它为参数补全、变量绑定和平台校验提供了可靠前提。第三，它让第三阶段在面对复杂动作时仍然保持可解释和可调试，而不是再次退回黑盒生成。

如果后续要把第三阶段真正做成可执行编译器，节点映射系统就是最先需要稳定下来的核心模块之一。
