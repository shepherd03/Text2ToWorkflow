# 第三阶段现状与解决方案汇报报告

## 一、汇报目的

这份报告用于组会汇报，聚焦第三阶段“DSL 落地”目前真实存在的困难、这些困难为什么会出现，以及现阶段准备采用什么办法推进。报告重点放在技术问题本身和可执行的解决路径上，不再重复铺陈系统价值判断，也不展开讨论前两阶段的职责调整。

## 二、当前项目进展

目前项目已经完成前两阶段的基础建设，并形成了相对稳定的输入输出链路。

第一阶段“UTR 生成”负责把自然语言任务描述转化为统一任务表示。当前能够输出任务目标、核心动作、核心资源、核心变量以及动作之间的隐式依赖关系。这个阶段的输出是元数据层，不包含控制流。

第二阶段“骨架规划”负责在 UTR 基础上生成抽象骨架树。当前骨架树已经能表达五类基础节点：`ActionSlot`、`SequentialBlock`、`ParallelBlock`、`ConditionalBlock`、`LoopBlock`。骨架规划采用“算法主导拓扑，大模型辅助判断”的方式，已经能够在基础拓扑结构上补入条件和循环等控制语义。

从现有状态看，系统已经能把自然语言任务转换为结构化抽象结果，第三阶段面对的不是“任务理解失败”，而是“如何把现有抽象结果进一步翻译为 Dify Workflow 可执行 DSL”。

## 三、当前面临的问题

第三阶段的困难并不集中在某一个点上，而是分布在“节点类型、参数生成、结构翻译、变量绑定、平台校验”几个层面。它们彼此相关，任何一个层面处理不稳，最终生成的 DSL 都可能无法导入或无法运行。

### 3.1 抽象节点到平台节点的映射还没有建立

第二阶段输出的是抽象骨架节点，但 Dify 实际运行的是平台节点。两者之间并不是一一对应关系。

例如，`ActionSlot` 只是表明“这里有一个动作要执行”，但没有说明这个动作在 Dify 上应该落成 `llm`、`code`、`template-transform`、`http-request` 还是其他节点。`ConditionalBlock` 和 `LoopBlock` 也只是表达控制意图，并没有直接给出 Dify 所要求的具体字段结构。

这带来两个直接问题：

第一，当前骨架结果不能直接拿去做 DSL 序列化，因为节点类型还没有定型。

第二，同一个动作在不同任务上下文里，落地节点类型可能不同。例如“总结内容”大概率对应 `llm`，但“格式化输出”有时更适合 `template-transform` 或 `code`。如果没有一层专门的类型决策逻辑，后续参数生成和变量绑定都无法开始。

### 3.2 现有动作信息不足以直接生成 Dify 节点配置

目前第一阶段动作对象包含的主要字段是 `action_name`、`description`、`inputs`、`outputs`。这些字段足够支撑前两阶段，但还不够直接支撑平台级落地。

第三阶段真正需要处理的是节点配置，而节点配置通常比动作元数据细得多。例如：

- `llm` 节点至少需要模型策略、prompt 结构、上下文引用方式和输出字段；
- `http-request` 节点至少需要 URL、method、headers、query/body 参数绑定；
- `tool` 节点不仅要知道调用工具，还要知道 provider、tool_name、参数结构；
- `iteration` 节点需要确定迭代输入来自哪里、循环体起点是什么、循环输出如何汇总；
- `if-else` 节点需要把自然语言条件转成平台接受的结构化条件表达式。

也就是说，第三阶段要处理的不是“有没有动作”，而是“如何把动作补全成一个可以运行的节点对象”。

### 3.3 树结构到图结构的转换难度比表面看起来更大

第二阶段输出的是树，Dify 需要的是图。这个问题不只是数据格式变化，而是控制流语义的重新展开。

顺序块相对简单，基本可以展开为一条链。真正复杂的是另外三类结构。

并行块的问题在于分支怎么汇合。骨架树里可以自然地表示“多个分支并行执行后继续往下走”，但在 Dify 图结构里，后续节点是否支持多个前驱、多个分支的输出如何统一、是否需要显式 barrier 或聚合节点，都要在第三阶段明确决定。

条件块的问题在于分支条件如何表达。当前骨架里保存的是自然语言条件描述，但 Dify `if-else` 需要的是比较具体的条件配置，比如变量选择器、比较算子、比较值、变量类型等。也就是说，第三阶段不仅要建分支，还要把条件文本转成结构化判断。

循环块的问题在于迭代容器怎么落地。骨架树里一个 `LoopBlock` 就足以表达“这里需要循环”，但 Dify 实际需要的是 `iteration` 容器、内部起始节点、迭代输入选择器、并行参数、输出选择器等一整套字段。循环语义如果解释不清楚，很容易出现看起来有循环、实际上平台无法运行的情况。

### 3.4 变量流转和引用绑定现在还缺少统一机制

前两阶段已经识别出了变量和动作依赖，但第三阶段需要的是平台级变量引用机制。这里涉及的不只是变量名，而是“谁产生变量、变量以什么字段输出、下游节点如何精确引用、并行或循环场景下如何汇总”。

这一部分如果没有统一规则，会出现几个典型问题：

- 上游节点虽然有输出，但下游节点无法确定引用路径；
- 同名变量在不同控制域里发生覆盖或歧义；
- 并行分支的多个输出无法被后续节点统一消费；
- 循环体内部变量和循环外部变量作用域混淆；
- 开始节点需要用户输入的变量和运行中产生的中间变量混在一起。

第三阶段如果只关注节点，不建立变量解析与绑定机制，最终生成的 DSL 往往会在运行时失败，而不是在生成时暴露问题。

### 3.5 兼容性问题必须在生成过程中持续暴露

第三阶段不是把 YAML 写出来就算完成。真正的目标是生成可导入、可执行、可调试的 Dify Workflow。

这意味着很多错误不能等到最后再看。例如：

- 节点类型是否在 Workflow 模式白名单内；
- `start` 和 `end` 是否各且仅有一个；
- 边的 source 和 target 是否都存在；
- `if-else` 是否至少有一条有效分支；
- `iteration` 是否拥有合法的内部起点和输出定义；
- 变量选择器是否指向真实存在的节点字段；
- 图中是否存在悬挂节点、孤立边或非法环。

如果这些问题不在中间阶段持续检查，等到 YAML 输出后再回头找原因，排查成本会很高，而且错误位置往往已经偏离真正问题源头。

## 四、当前预想的解决方案

现阶段准备把第三阶段实现为一个“面向 Dify Workflow 的结构化编译过程”，而不是做成简单的 YAML 拼接器。整体思路是先把抽象骨架翻译成内部可推理结构，再逐步渲染成平台 DSL。

### 4.1 先建立分步骤的编译链路

目前预想的主链路分为八步：

1. 输入接入与前置校验；
2. 骨架归一化与结构补全；
3. `ActionSlot` 节点类型决策；
4. 控制流图级翻译；
5. 变量选择与引用绑定；
6. 图节点、图边与布局生成；
7. Dify 平台兼容校验；
8. DSL 序列化与持久化。

这样拆开的原因很直接：第三阶段的问题本来就不是一个步骤能解决的。节点类型、变量绑定、图结构和平台校验必须拆开处理，否则逻辑会很快纠缠在一起，后续调试也会失控。

### 4.2 用规则优先的方式完成节点类型定型

在节点映射问题上，当前不打算让模型直接自由决定全部节点类型，而是先使用规则做初筛，再在必要时用 LLM 补充判断。

一个可行的做法是把动作按照语义关键词和输入输出特征做主映射：

- 生成、摘要、改写、分析类动作优先映射为 `llm`；
- 数据转换、清洗、合并、计算类动作优先映射为 `code`；
- 文本拼装、模板填充类动作优先映射为 `template-transform`；
- 外部接口调用类动作优先映射为 `http-request`；
- 分支判断由 `if-else` 承接；
- 批量遍历由 `iteration` 承接。

如果规则命中不明显，再让模型在有限候选集中做二选一或多选一判断，而不是开放式生成。这样做可以把模型的作用压缩在“辅助分类”这个局部任务上，避免第三阶段重新退回黑盒生成。

### 4.3 用控制流降级策略处理树到图的转换

第三阶段不直接保留抽象块，而是把每个控制块编译成“具有入口和出口的子图”。

顺序块的处理方式最明确：按顺序连接前一子图出口和后一子图入口即可。

并行块的处理重点放在汇合上。当前更稳妥的做法不是依赖平台默认的多前驱行为，而是预留显式汇合策略：

- 如果下游只关心控制顺序，不直接消费多分支数据，可以考虑用轻量 barrier 方式汇合；
- 如果下游要显式消费多个分支输出，则插入 `variable-aggregator` 或等价聚合节点；
- 对于平台行为不明确的情况，优先采用更保守的显式汇合，而不是假设平台一定能自动等待全部前驱完成。

条件块则统一围绕 `if-else` 节点展开。核心不是做分支，而是条件结构化。对于条件描述，当前准备分三层处理：

1. 规则能直接识别的条件，直接转成结构化配置；
2. 规则无法识别的条件，调用 LLM 输出受约束 JSON；
3. 如果仍然无法可靠解析，则保留待补全状态，而不是生成错误条件。

循环块统一围绕 `iteration` 实现。关键工作不在“是否有循环”，而在“确定迭代输入、内部起点、输出汇总”。这里会优先从变量类型、动作输出和循环描述中的关键词里反推 iterator 来源，只有定位到可用输入后才渲染标准 iteration 结构。

### 4.4 建立统一的变量解析和引用规则

变量问题不能附带处理，需要单独建立规则。

当前准备把变量分成三类：

- 由开始节点提供的输入变量；
- 由业务节点中间产生的过程变量；
- 在并行或循环中聚合得到的汇总变量。

基于这个划分，第三阶段需要做四件事：

第一，确定哪些 `core_variables` 和 `core_resources` 要暴露为 `start` 节点输入。

第二，为每个业务节点定义稳定输出字段。哪怕早期只约定 `text`、`result`、`items` 这类保守字段，也必须统一，否则下游无法可靠引用。

第三，把节点间引用抽象为内部选择器对象，在最终渲染时再转换成 Dify 的 `value_selector` 或模板表达式。

第四，对并行和循环结果明确聚合出口，避免多个分支或多轮迭代直接把结果散落给后续节点。

这套机制一旦建立，第三阶段就不只是“连节点”，而是能真正维护一条可追踪的数据流。

### 4.5 把参数补全控制在可验证范围内

第三阶段确实需要补全参数，但补全不能失控。

当前准备采用明确的优先级：

1. 先用 UTR 已有字段；
2. 再用规则推导默认值；
3. 再用预制节点模板补齐常见字段；
4. 规则仍不足时，再让 LLM 生成结构化候选；
5. 如果仍无法满足平台要求，则明确输出“信息不足”或“待补全”状态。

这个顺序有两个目的。一个是尽量把生成建立在确定性信息上，减少不必要的模型猜测；另一个是避免为了追求“看起来生成成功”而伪造并不存在的 URL、工具配置或循环参数。

### 4.6 在中间阶段持续做平台校验

第三阶段准备把校验前移到中间表示层，而不是放在 YAML 输出之后。

校验至少分成四层：

1. Schema 层：检查内部对象字段是否完整、类型是否正确；
2. 图结构层：检查节点唯一性、边合法性、入口出口是否完整；
3. 平台约束层：检查节点白名单、开始结束节点、iteration 结构、if-else 配置等；
4. 引用绑定层：检查变量选择器是否有效、字段来源是否存在、跨作用域引用是否合理。

这样做以后，即使最终 DSL 失败，也能比较明确地定位到失败是出在节点类型、结构降级、变量绑定还是平台兼容，而不是只拿到一个笼统的“导入失败”。

## 五、下一步实施安排

当前不适合一开始就追求全覆盖，更合理的推进方式是先做最小可运行闭环，再逐步扩展复杂控制流和更丰富的节点类型。

### 5.1 先打通最小可运行链路

第一步先实现 `start + code/llm + end` 的顺序工作流生成。这个阶段的目标不是覆盖更多场景，而是确认第三阶段最核心的编译主干是否跑通，包括：

- 输入校验是否稳定；
- 节点注册机制是否清晰；
- 边生成是否正确；
- 变量输入输出是否有统一约定；
- YAML 序列化后的结构是否能通过基础校验。

### 5.2 再处理并行、条件、循环

最小闭环稳定后，再逐步引入并行、条件和循环。这样可以把问题拆开处理：

- 并行重点验证显式汇合策略是否稳定；
- 条件重点验证自然语言条件能否转成结构化条件；
- 循环重点验证 iterator 识别与 iteration 容器生成是否可靠。

如果这三类控制结构一开始同时推进，排错时很难分清到底是映射问题、图问题还是变量问题。

### 5.3 逐步提升节点语义精度

在早期版本里，适当使用 `code` 或 `llm` 作为保守降级节点是可接受的，因为它有助于先建立闭环。但闭环形成后，需要逐步把常见动作提升为更准确的平台节点，例如：

- 外部 API 调用提升为 `http-request`；
- 模板拼接提升为 `template-transform`；
- 多分支汇总提升为 `variable-aggregator`。

这样做的意义在于，后期不是只让 DSL “能跑”，而是让它更符合 Dify 原生工作流表达方式。

### 5.4 用失败样例驱动后续细化

第三阶段开发过程中，需要同步保留失败样例，重点记录三类信息：

- 哪些任务在节点类型判断上不稳定；
- 哪些任务在条件或循环解释上缺失关键信息；
- 哪些任务在图结构或变量绑定上反复失败。

这部分记录一方面可以帮助后续调优第三阶段规则，另一方面也能为以后是否需要增加可选提示字段提供证据。但在现阶段，这些记录首先服务于第三阶段自身完善，而不是提前引出架构调整。
