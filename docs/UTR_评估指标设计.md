# UTR 生成效果评估指标设计方案

本文档定义了如何量化评估大模型生成的 Unified Task Representation (UTR) 的质量。我们将生成的 UTR 与从 Dify DSL 提取的“标准 UTR (Ground Truth)”进行对比，计算综合得分。

## 1. 评估体系概览

评估采用 **满分 100 分制**，分为四个核心维度。每个维度独立打分，最后加权求和。

| 维度 | 权重 | 核心关注点 | 评估方法 |
| :--- | :--- | :--- | :--- |
| **动作准确性 (Action Accuracy)** | **40%** | 是否正确识别了所有核心步骤？ | 基于动作名称和工具类型的 F1 Score |
| **参数完整性 (Parameter Completeness)** | **30%** | 动作所需的资源和变量是否齐全？ | 关键参数的覆盖率与值的准确性 |
| **逻辑连贯性 (Logic Coherence)** | **20%** | 执行顺序、条件判断、循环逻辑是否正确？ | 控制流类型匹配度与条件语义相似度 |
| **结构合法性 (Schema Validity)** | **10%** | 输出是否符合 UTR JSON Schema 定义？ | JSON 校验通过率 |

---

## 2. 详细评分规则

### 2.1 维度一：动作准确性 (Action Accuracy) - 40分

**目标**：衡量模型是否“找对”了要做的事情。这是工作流执行的基础。

**计算逻辑**：
采用 **F1 Score** 算法，综合考虑精确率 (Precision) 和召回率 (Recall)。

*   **TP (True Positive)**: 模型生成的动作在标准答案中存在（且位置/顺序基本一致）。
*   **FP (False Positive)**: 模型生成了多余的动作（幻觉）。
*   **FN (False Negative)**: 模型漏掉了标准答案中的动作。

**匹配标准**：
*   **强匹配**：`tool_name` (工具名) 必须完全一致。
*   **弱匹配**：`action_name` (动作名) 允许语义相似（如 "send_email" vs "send_mail"），若使用 LLM 辅助判断语义一致则视为匹配。

**得分公式**：
$$ Score_{Action} = 2 \times \frac{Precision \times Recall}{Precision + Recall} \times 40 $$

> **示例**：
> *   标准答案：[搜索天气, 发送邮件]
> *   模型输出：[搜索天气, 生成报告, 发送邮件]
> *   分析：TP=2 (搜索天气, 发送邮件), FP=1 (生成报告), FN=0
> *   Precision = 2/3 = 0.67
> *   Recall = 2/2 = 1.0
> *   F1 = 0.8 -> 得分 = 0.8 * 40 = 32 分

### 2.2 维度二：参数完整性 (Parameter Completeness) - 30分

**目标**：衡量动作执行所需的“原料”是否准备充足。

**计算逻辑**：
针对每一个匹配成功的动作（TP），检查其关联的 `args` (参数)、`resources` (资源) 和 `variables` (变量)。

*   **关键参数匹配**：
    *   **名称 (Name)**：参数名是否一致？（权重 0.4）
    *   **值 (Value)**：参数值是否一致？对于变量引用（如 `{{weather}}`），检查引用关系是否正确。（权重 0.6）

**得分公式**：
$$ Score_{Param} = (\frac{\sum \text{每个匹配动作的参数得分}}{\text{匹配动作总数}}) \times 30 $$

> **特殊情况处理**：
> *   如果模型输出了标准答案中没有的参数（但在合理范围内，如默认值），不扣分。
> *   如果关键参数（如 API Key, 目标邮箱）缺失，该动作参数分直接为 0。

### 2.3 维度三：逻辑连贯性 (Logic Coherence) - 20分

**目标**：衡量工作流的“流程图”画得对不对。

**计算逻辑**：
检查 `control_intents` 列表。

1.  **类型匹配 (Type Match)**：
    *   `Sequential` (顺序): 基础分。
    *   `Conditional` (条件): 重点检查 `condition` 表达式。
    *   `Iteration` (循环): 重点检查 `loop_target`。

2.  **条件语义匹配**：
    *   标准答案可能写的是代码逻辑 `result.code == 200`。
    *   模型可能生成自然语言逻辑 `请求成功`。
    *   **评分策略**：使用轻量级 LLM 或语义相似度算法判断两者是否等价。等价则得分。

**得分公式**：
$$ Score_{Logic} = (\text{类型匹配率} \times 0.5 + \text{条件/循环目标语义匹配率} \times 0.5) \times 20 $$

### 2.4 维度四：结构合法性 (Schema Validity) - 10分

**目标**：基础门槛，格式不对无法被系统解析。

**计算逻辑**：
*   **JSON 解析成功**：得 2 分。
*   **必需字段完整** (actions, resources 等顶层字段存在)：得 3 分。
*   **字段类型正确** (如 actions 是 list 不是 dict)：得 3 分。
*   **枚举值合法** (如 type 必须在 schema 定义的 Enum 中)：得 2 分。

**得分公式**：
$$ Score_{Schema} = \text{通过校验项分数之和} $$

## 3. 综合评分示例

假设一个“查询天气并发送”的任务：

| 维度 | 评估结果 | 原始得分 | 加权后得分 |
| :--- | :--- | :--- | :--- |
| **动作** | 找对了所有步骤，没多没少 | 1.0 | **40** |
| **参数** | 邮箱地址填对了，但城市参数漏了 | 0.5 | **15** |
| **逻辑** | 这是一个顺序任务，模型识别为顺序 | 1.0 | **20** |
| **结构** | 完美符合 Schema | 1.0 | **10** |
| **总分** | | | **85 / 100** |
