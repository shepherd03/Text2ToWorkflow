# 第三阶段 DSL 落地规划报告

## 一、报告定位与目标

本报告用于指导统一任务表示（UTR）工作流自动生成系统的第三阶段建设，即“DSL 落地”阶段。该阶段的任务不是重新理解用户需求，也不是重新规划工作流控制结构，而是在严格遵守现有三阶段流水线架构的前提下，将第二阶段生成的抽象骨架树稳定、可控、可验证地转换为符合 Dify Workflow 规范的 DSL。报告的目标是为后续逐步开发提供一份可直接执行的总方案，尽可能把方法、边界、模块设计、兼容策略、验证方案、风险点和实施路线一次性讲清楚，避免后续开发过程中反复推翻。

第三阶段必须坚持以下基本立场。第一，不能破坏项目“分层解耦、逐步求精”的核心路线，阶段三只负责“把已经抽象好的结构落地成平台可运行表达”，不能反向侵入前两阶段的职责。第二，不能把 Dify DSL 生成完全交给大模型自由发挥，最终输出必须由规则、模板、映射关系和程序化校验共同决定，大模型最多只能作为局部语义补全器。第三，输出目标必须优先保证 Dify Workflow 的兼容性和可执行性，而不是追求表面上的“像 YAML”。第四，第三阶段不是简单字符串拼接，而是一次“抽象结构到平台图结构”的编译过程，因此需要引入明确的中间表示、节点映射规则、变量绑定规则和图校验规则。

从系统整体演化看，第三阶段的价值体现在三个层面。其一，它把现有项目从“能分析、能规划”推进到“能运行、能落地”；其二，它会反向暴露前两阶段输出中哪些信息不足，从而为系统后续完善提供明确反馈；其三，它会建立一套平台兼容约束，使整个工作流生成系统不再停留在抽象研究阶段，而是具备工程闭环。

## 二、第三阶段在全局架构中的职责边界

为了避免阶段边界混乱，必须先把第三阶段的输入、输出和禁止事项讲清楚。

第三阶段的**直接输入**应当是第二阶段生成的骨架树，必要时辅以原始 UTR 作为补充语义来源。骨架树提供控制流主干，UTR 提供动作、变量、资源和依赖元数据。也就是说，第三阶段的输入不是自然语言任务，也不是未结构化文本，而是“已经被前两阶段约束过的结构化对象”。

第三阶段的**核心输出**应当是一个满足 Dify Workflow 规范的 DSL 文档对象，最终可序列化为 YAML 或 JSON，并能够在 Dify 平台中导入、执行、调试。这个输出至少要包含以下内容：顶层 app 信息、workflow 基本配置、graph.nodes、graph.edges、必要的变量定义、开始节点、结束节点，以及具体业务节点的 data 配置。

第三阶段**明确不负责**以下事情：第一，不重新从自然语言决定业务步骤；第二，不重新设计整体拓扑结构；第三，不在无依据的情况下发明不受 Dify 支持的节点类型；第四，不修改前两阶段的核心接口签名；第五，不把平台配置硬编码成一次性实验脚本。

因此，第三阶段更准确的角色应该被定义为：**一个面向 Dify Workflow 的结构化编译器**。它读取“UTR + Skeleton”这一抽象程序，输出“Dify DSL”这一平台程序。

## 三、第三阶段面临的核心问题

如果不先看清问题，后续方案容易失焦。结合现有第一阶段、第二阶段实现方式，以及 Dify Workflow 规范，第三阶段至少面临六个本质问题。

### 3.1 抽象骨架与平台节点之间存在语义鸿沟

第二阶段骨架树中的节点类型是抽象控制节点，例如 `SequentialBlock`、`ParallelBlock`、`ConditionalBlock`、`LoopBlock` 和 `ActionSlot`。但 Dify Workflow 中真正能运行的是 `start`、`end`、`llm`、`tool`、`if-else`、`iteration`、`code`、`http-request`、`template-transform` 等具体平台节点。也就是说，阶段二告诉我们“这里有一个动作槽位”，但没有告诉我们“它在 Dify 里应该用哪个节点表达”。第三阶段必须解决这个类型落地问题。

### 3.2 现有动作元数据不足以直接生成完整节点参数

UTR 中的 `Action` 目前只有 `action_name`、`description`、`inputs`、`outputs`。这些字段足以支持前两阶段的抽象规划，但不足以直接生成 Dify 节点。例如 `llm` 节点需要模型配置和 prompt 模板，`http-request` 节点需要 URL、method、headers，`tool` 节点需要 provider 信息和参数结构。换句话说，第三阶段不能假设前两阶段已经提供了完整平台参数，它必须设计一套“语义补全与默认落地”的方法。

### 3.3 Dify Workflow 对控制流的表达不是抽象树，而是图

第二阶段输出的是树形骨架，Dify 落地时真正需要的是图结构：节点列表与边列表。树和图不是一回事。顺序块可以较容易映射为链式边，条件分支可以映射为 `if-else`，循环可以映射为 `iteration`，但并行块在图结构中常常还涉及分叉、汇合、变量聚合与后继依赖等待问题。第三阶段必须把“树上的控制语义”准确地变成“图上的执行语义”。

### 3.4 Dify 平台没有“纯抽象占位节点”

在内部编译过程中我们可能需要 join、barrier、branch-exit 等中间概念，但 Dify 不提供这些抽象节点。因此第三阶段需要一种机制，把内部临时控制语义降级为 Dify 支持的真实节点，或者在不需要真实节点时直接用边关系完成表达。

### 3.5 兼容性校验不能放到最后才做

如果到了序列化成 YAML 时才发现某些字段缺失、变量引用非法、循环体不合法、某节点类型无法放入 Workflow 模式，代价会非常高。第三阶段必须采用“边生成边校验”的策略，把 Dify 兼容约束前移到中间表示层。

### 3.6 第三阶段必须为后续迭代留下扩展位

当前目标是聚焦 Dify Workflow 场景，而不是一次性覆盖 Agent 高级编排及所有节点能力。方案必须允许我们先实现一个高可用的最小闭环，再逐步扩展节点覆盖率和高级能力，而不是一开始把所有复杂功能压在一个单体生成器里。

## 四、第三阶段的总体设计原则

基于以上问题，第三阶段建议遵循以下七条设计原则。

### 4.1 编译器思维优先，模板拼接思维靠后

第三阶段的本质不是“写 YAML”，而是“编译工作流”。因此实现上要分层处理：输入解析、结构归一化、节点类型决策、图构建、参数填充、引用绑定、布局计算、平台校验、序列化输出。只有这样，后续扩展、测试和调试才可控。

### 4.2 算法主导生成，模型辅助补全

延续项目前两阶段的基本哲学，DSL 生成也不能完全黑盒。控制流翻译、节点连线、变量引用、节点 ID 生成、布局坐标、拓扑合法性、字段结构校验必须由程序规则完成。大模型只在两个局部场景中参与：第一，判断 `ActionSlot` 更适合映射为哪类 Dify 节点；第二，在缺失参数时生成受约束的候选模板，例如 LLM 节点的 prompt 模板草案或条件节点的 case 文本解释。即使使用大模型，也必须返回结构化 JSON，再由程序端进行验证和裁剪。

### 4.3 聚焦 Dify Workflow 单目标

根据你的最终目标，第三阶段必须只面向 Dify Workflow 的导入与运行来设计。这样可以把资源集中到开始节点、结束节点、业务节点和控制流结构的落地上，避免把实现精力分散到与当前目标无关的交互式对话能力上。也就是说，阶段三的所有模型设计、控制流翻译、变量绑定和校验规则，都应直接围绕 Dify Workflow 的可运行性展开，而不是为多种运行模式预留复杂分支。

### 4.4 引入中间表示，隔离骨架树与平台 DSL

建议在阶段三内部引入至少一层中间表示（IR），把“骨架级别结构”转换为“DSL 级别可渲染结构”。这样做好处非常大：一是可以在不碰 YAML 的情况下做大量逻辑测试；二是可以把平台相关校验集中在 IR 到 DSL 的最后一步；三是后续如果目标平台扩展为其他工作流系统，仍可复用大部分前置逻辑。

### 4.5 最小可运行优先于一次性完美覆盖

第三阶段应当先实现一个能真实导入 Dify、能跑通简单顺序、并行、条件、循环的最小可运行版本，然后逐步补足更多节点类型与高级配置。不要一开始就试图覆盖所有 Dify 节点，这会导致设计膨胀、测试失控和接口不稳定。

### 4.6 平台兼容性前置

任何一个内部结构只要无法稳定降级为 Dify Workflow 支持的节点和边，就不应该进入最终输出环节。第三阶段应当构建一套“平台兼容 profile”，明确哪些字段是必填、哪些节点在 Workflow 模式可用、哪些控制流需要特殊降级策略。

### 4.7 输出必须可验证、可回溯、可解释

生成后的 DSL 不是“黑箱产物”，必须能解释每个节点是从哪个骨架节点、哪个动作、哪条依赖、哪类映射规则得到的。建议在内部保留 trace 信息，并在生成元数据中记录映射来源，便于调试与评估。

## 五、建议采用的第三阶段总流程

第三阶段建议拆成九个连续步骤，每一步都有明确输入和输出，不建议跳步。

### 5.1 第一步：输入接入与前置校验

输入对象为 `UTR + Skeleton`。在这一步需要完成：

1. 校验骨架树是否符合 Schema；
2. 校验骨架树中是否只使用允许的五类节点；
3. 校验所有 `ActionSlot.action_id` 是否能在 `UTR.metadata.core_actions` 中找到；
4. 校验骨架树不存在空分支、空循环体、非法嵌套；
5. 若骨架树不合法，直接停止，不进入 DSL 生成。

这一步输出的是“通过前置校验的骨架上下文对象”。

### 5.2 第二步：归一化与结构补全

现有第二阶段生成的骨架树中已经人为加入了 `start_node` 与 `end_node` 这两个动作槽位，但它们目前仍是抽象 `ActionSlot`。在 DSL 落地时，需要把它们视为真正的平台开始节点与结束节点。与此同时，还要把一些隐式信息补全出来，例如：

- 根节点的唯一入口与唯一出口；
- 每个块的父子关系；
- 每个动作槽位的上游、下游、所在控制域；
- 并行块是否需要汇合；
- 条件块是否存在默认 `else` 分支；
- 循环块的循环体入口节点与循环出口节点。

建议在这一阶段生成一棵“标准化骨架树”，消除不同生成路径导致的结构差异。

### 5.3 第三步：ActionSlot 语义定型

这是阶段三最关键的一步。抽象动作必须先决定“用什么 Dify 节点落地”，否则后续所有字段都无从谈起。建议建立一套 **节点类型决策器**，按照“规则优先、LLM 补充”的方式工作。

规则层可根据 `action_name`、`description`、`inputs`、`outputs`、资源类型和关键词进行初步归类，例如：

- 包含 `extract`、`parse`、`classify`，优先考虑 `parameter-extractor`、`question-classifier`；
- 包含 `generate`、`summarize`、`rewrite`、`analyze`，优先考虑 `llm`；
- 包含 `request`、`fetch_api`、`post_data`，优先考虑 `http-request`；
- 包含 `execute_script`、`transform_json`、`merge_results`，优先考虑 `code`；
- 包含 `call_tool`、`search_web`，优先考虑 `tool`；
- 包含 `fill_template`、`compose_text`，优先考虑 `template-transform`；
- 包含 `extract_document`、`read_file_text`，优先考虑 `doc-extractor`。

若规则层无法确定，则调用受约束的 LLM 分类器，只允许从白名单节点类型中选择，并输出理由、置信度和候选参数草案。程序端再根据置信度决定是否接受。如果仍无法确定，则统一回退为 `code` 或 `llm` 的保守实现，但必须在生成元数据中打标，提示该节点为降级节点。

这一步的输出不再是抽象 `ActionSlot`，而是一个“带平台类型的动作节点描述对象”。

### 5.4 第四步：控制流降级

这一步把骨架树的五类节点翻译成 Dify 图结构可接受的控制表达。

#### 5.4.1 顺序块降级

`SequentialBlock` 最简单，按子节点顺序依次建立前驱与后继关系即可。它本质上会被翻译为一条或多条连续边。

#### 5.4.2 并行块降级

`ParallelBlock` 需要重点设计。建议采用“分叉 + 分支子图 + 汇合策略”的办法。

- 分叉：前驱节点连接到每个分支的首节点；
- 分支内部：各自独立展开；
- 汇合：根据下游使用方式选择策略。

并行汇合至少有两种模式：

1. **纯控制汇合**：多个分支完成后，再继续执行同一个后续节点，但后续节点不需要直接消费多个分支产物。这种情况下可允许多个分支节点直接连向同一后继节点，前提是 Dify 对该节点支持多前驱执行等待；如果平台限制不稳定，建议插入一个内部 `JoinBridge`，最终降级为一个最小 `code` 节点作为 barrier。
2. **数据汇合**：后续节点需要显式使用多个分支输出。这时应优先生成 `variable-aggregator` 或 `code` 聚合节点，将多个来源整理成一个标准输出，再传给后续节点。

因此，并行块的实现不能只做“多个边同时出去”，必须同时定义汇合规则。

#### 5.4.3 条件块降级

`ConditionalBlock` 应当映射为 Dify 的 `if-else` 节点。当前骨架中的 `condition_description` 只是自然语言说明，因此需要增加一个 **条件表达式解析子步骤**：把自然语言条件转成 Dify `cases.conditions` 所需的字段，包括 `variable_selector`、`comparison_operator`、`value`、`varType`。

建议分三级处理：

1. 能由规则识别的条件直接结构化，例如“输入为空”“文件存在”“数量大于 10”；
2. 规则无法识别时，调用 LLM 将条件描述转换成受约束 JSON；
3. 若仍无法可靠解析，则降级为“人工待补全条件”模式，但不应生成不可导入 DSL。

此外，条件块至少要有 `true` 分支；如果没有显式 `false` 分支，程序端应自动补一个空 else 路径，保证 Dify 结构完整。

#### 5.4.4 循环块降级

`LoopBlock` 应当映射为 Dify 的 `iteration` + `iteration-start` 组合。这里是阶段三另一个难点，因为现有骨架中的 `loop_condition` 常常只是“遍历所有文件”“逐条处理数据”这类自然语言，而 Dify 的 `iteration` 节点需要更具体的信息，如：

- `iterator_selector`：循环输入来源；
- `is_array_input`；
- `start_node_id`；
- `output_selector`；
- `is_parallel` 和 `parallel_nums`。

因此，建议为循环块设计“循环意图解释器”，其任务不是重新生成循环结构，而是把已有循环语义补全成 Dify 所需配置。这个解释器可以从以下来源取信息：

1. UTR 变量中类型为 `list`、`file`、`data` 的候选变量；
2. 上游动作输出字段；
3. `loop_condition` 的关键词，如“遍历列表”“批量处理文件”“对每条记录执行”；
4. 必要时的 LLM 结构化补全。

若能够定位到迭代输入，则构建标准 `iteration` 容器；若无法定位，短期内可降级为顺序批处理 `code` 节点，但必须标记该循环为“未完全平台化”，避免误判成功。

### 5.5 第五步：变量与引用绑定

Dify DSL 的可运行性很大程度上取决于变量绑定是否正确。第三阶段必须建立统一的变量引用策略。

建议采用以下规则：

1. **开始节点输入变量**：从 UTR 的 `core_variables` 和 `core_resources` 中筛选出需要由用户输入提供的变量，在 `start.variables` 中定义；
2. **节点输出变量**：每个业务节点都要定义稳定的输出字段名，哪怕是内部统一约定的 `text`、`result`、`items`，也必须可预测；
3. **节点输入绑定**：后继节点引用前驱节点输出时，统一生成 `value_selector` 或 Mustache 模板；
4. **环境变量**：涉及 API 密钥、服务地址等敏感信息时，不写死到节点参数中，而是在 `workflow.environment_variables` 中占位；
5. **聚合变量**：并行或循环场景的汇总结果，必须通过聚合节点输出为一个确定字段，再供下游使用。

这一步最好把所有变量引用抽象为内部选择器对象，直到最后序列化时再转换为 Dify 的数组或模板形式。

### 5.6 第六步：图构建与布局生成

当节点类型和变量绑定都明确后，就可以构建 `workflow.graph.nodes` 与 `workflow.graph.edges`。这里建议不要直接在遍历骨架时临时拼节点，而是使用图构建器统一管理：

- 负责生成全局唯一节点 ID；
- 负责注册节点、边、父子关系；
- 负责处理 iteration 内部节点的 `parentId`、`extent`、`isInIteration`；
- 负责生成 edge 的 `sourceHandle`、`targetHandle` 与 data；
- 负责统一布局坐标。

布局不必一开始追求美观，但必须稳定、可读、可导入。建议采用“层次布局”：

- 顺序节点按 x 轴递增；
- 并行分支按 y 轴分叉；
- 条件分支左右或上下分叉；
- 迭代体内部节点在容器内单独排布。

只要布局稳定，后续用户在 Dify 画布中就可以继续手工微调。

### 5.7 第七步：平台 Profile 校验

在生成最终 DSL 之前，必须执行平台兼容校验。建议至少校验以下内容：

1. 顶层 `version` 是否为 `0.2.0`；
2. `kind` 是否为 `app`；
3. `app.mode` 是否为 `workflow`；
4. 所有节点 `data.type` 是否属于 Workflow 支持的白名单；
5. 开始节点和结束节点是否恰好各一个；
6. 每条边的 source/target 是否存在；
7. `iteration` 节点是否存在合法 `start_node_id`；
8. `if-else` 是否存在至少一个 case；
9. 所有变量引用选择器是否合法；
10. 整体图是否无悬挂节点、无孤立边、无非法环。

这一步应当返回结构化报告，而不是只抛异常。这样便于后续 API 输出给用户查看。

### 5.8 第八步：序列化输出

当 DSL 对象通过校验后，再统一序列化为 YAML。序列化阶段应当做到：

- 保持字段顺序稳定，方便 diff；
- 使用 UTF-8；
- 避免无意义的空字段；
- 对列表、字典保持规范缩进；
- 保留平台要求的布尔、数字、字符串类型，不做全部字符串化。

### 5.9 第九步：产物持久化与可追溯记录

按照项目既定规范，第三阶段输出也应采用 JSONL 持久化到 `generated_data/dsl_generation/` 子目录中。建议一条记录至少包含：

- `id`；
- `utr` 摘要；
- `skeleton` 摘要；
- `dsl_object`；
- `dsl_yaml`；
- `validation_report`；
- `generation_meta`。

这样后续评估与回归测试才有基础。

## 六、建议引入的内部中间表示（IR）

为了让第三阶段实现真正可维护，强烈建议引入两层 IR，而不是直接从骨架树生成 YAML。

### 6.1 第一层：DSLPlanningIR

这层用于表达“已经确定平台节点类型，但尚未完全渲染为 Dify 字段”的结构，建议包含：

- 规划节点：`id`、`source_action_id`、`node_kind`、`title`、`input_specs`、`output_specs`、`config_stub`；
- 控制节点：`if`、`iteration`、`join`、`barrier`；
- 连接关系：逻辑边、分支标签、汇合标记；
- 作用域信息：是否在循环中、是否在条件分支中。

这层 IR 主要服务于“结构推理”和“规则决策”。

### 6.2 第二层：DifyGraphIR

这层已经接近最终 DSL，主要包含：

- `app`；
- `workflow.environment_variables`；
- `workflow.conversation_variables`（Workflow 模式通常为空）；
- `workflow.features`；
- `workflow.graph.nodes`；
- `workflow.graph.edges`。

这层 IR 主要服务于平台校验与序列化。

这种两层设计的好处是：前一层更抽象，方便编译逻辑；后一层更贴近平台，方便导出和校验。未来如果平台规范变化，只需要调整第二层渲染器，而不必重写整个阶段三。

## 七、面向 Dify Workflow 的节点映射策略

第三阶段不可能一开始支持所有 Dify 节点，因此建议建立“分批覆盖”的策略。第一批优先支持最常见、最能形成闭环的节点。

### 7.1 第一批必须支持的节点

1. `start`
2. `end`
3. `llm`
4. `code`
5. `if-else`
6. `iteration`
7. `iteration-start`
8. `template-transform`
9. `http-request`
10. `variable-aggregator`

这一批足以覆盖大量内容生成、信息处理、批处理和简单外部调用场景。

### 7.2 第二批建议支持的节点

1. `tool`
2. `parameter-extractor`
3. `question-classifier`
4. `knowledge-retrieval`
5. `doc-extractor`
6. `assigner`

这批节点会显著提高任务覆盖度，但参数复杂度更高，适合在第一批闭环稳定后推进。

### 7.3 动作到节点的建议映射表

建议不是直接硬编码一一映射，而是采用“主映射 + 降级映射”双层机制：

- 信息生成、摘要、改写、分析：主映射 `llm`，降级 `code`
- 文本模板拼接：主映射 `template-transform`，降级 `code`
- 外部 API 调用：主映射 `http-request`，降级 `code`
- 规则转换、数据清洗、汇总：主映射 `code`
- 条件判断：主映射 `if-else`
- 批量遍历：主映射 `iteration`
- 多来源汇总：主映射 `variable-aggregator`，降级 `code`
- 工具调用：主映射 `tool`，降级 `http-request` 或 `code`
- 文档解析：主映射 `doc-extractor`，降级 `code`

所谓降级映射并不是偷懒，而是为了保证最小闭环：即使还没有对某个高级节点做完全部参数建模，也可以先用 `code` 节点生成一个可运行的近似实现。

在不破坏现有目录规范的前提下，第三阶段建议新增如下模块：

- `src/dsl_generation/__init__.py`
- `src/dsl_generation/pipeline.py`
- `src/dsl_generation/dsl_generator.py`
- `src/dsl_generation/node_mapper.py`
- `src/dsl_generation/control_flow_lowering.py`
- `src/dsl_generation/variable_resolver.py`
- `src/dsl_generation/graph_builder.py`

其中要特别注意项目既有规则：**全局数据模型统一定义在 `src/core/schema.py`**。因此第三阶段新增的 IR、验证报告、DSL 输出模型，建议放到 `src/core/schema.py` 中统一定义，而不是在 `dsl_generation` 模块内各自再定义一套 Pydantic 模型。

推荐职责分配如下：

- `pipeline.py`：对外统一调用入口；
- `dsl_generator.py`：总调度器，串联所有步骤；
- `node_mapper.py`：动作到平台节点的类型映射与参数草案生成；
- `control_flow_lowering.py`：顺序、并行、条件、循环的图级翻译；
- `variable_resolver.py`：变量选择器和节点间引用绑定；
- `graph_builder.py`：节点、边、坐标与容器结构生成；

这样拆分后，每个模块都只做一件事，后续测试也更容易编写。

## 九、与现有 Schema 的兼容扩展建议

第三阶段实现过程中，必然需要补充新的数据模型，但必须遵守兼容原则，不破坏现有核心方法入参与出参。建议采用“新增可选模型和可选字段”的方式。

建议在 `src/core/schema.py` 中新增以下几类模型：

1. `DSLNodeSpec`：描述一个待渲染节点；
2. `DSLEdgeSpec`：描述一条边；
3. `DifyWorkflowGraph`：描述 graph；
4. `DifyWorkflowDSL`：描述最终 DSL 对象；
5. `DSLGenerationReport`：描述生成与校验报告；
6. `DSLGenerationOutput`：打包 DSL、报告和元数据。

此外，可以考虑对现有骨架模型做**向后兼容的可选增强**，例如：

- `ConditionalBlock` 增加 `condition_hint_variable`、`comparison_operator_hint`；
- `LoopBlock` 增加 `iterator_hint`、`is_parallel_hint`；
- `Action` 增加 `preferred_node_type` 或 `action_category` 的可选字段。

这些字段都必须是 optional，不影响现有阶段一、阶段二运行。这样既不破坏兼容性，又能为阶段三减少猜测成本。

## 十、详细方法：控制流翻译与平台落地

这一部分是整个规划的技术核心。

### 10.1 顺序控制流

顺序控制流最适合作为第一批实现目标。建议把一个顺序块编译为“首节点 + 中间节点链 + 末节点”，边关系按顺序串联。若顺序块内部包含条件、循环、并行，则在展开时把它们视为子图，最终返回该子图的入口节点集合与出口节点集合。这样，上层顺序块只需要把前一个子图的出口接到后一个子图的入口。

换句话说，第三阶段在编译每个控制块时，建议统一返回：

- `entry_nodes`
- `exit_nodes`
- `created_nodes`
- `created_edges`

这样不同控制块之间才容易组合。

### 10.2 并行控制流

并行控制流的难点不是“展开分支”，而是“分支之后如何继续”。建议采用子图接口法：

- 对每个分支单独编译，得到各自入口和出口；
- 并行块的入口集合为所有分支入口；
- 并行块的出口集合为所有分支出口；
- 若上层需要顺序连接到下一个节点，则调用 `merge_exits` 策略决定是否插入显式汇合节点。

`merge_exits` 建议至少支持三种策略：

1. `direct_join`：多分支直接连向下游；
2. `aggregator_join`：插入 `variable-aggregator`；
3. `code_barrier_join`：插入轻量 `code` 节点作为汇合器。

默认策略建议采用 `code_barrier_join`，因为它最稳妥，最不依赖特定下游节点行为。

### 10.3 条件控制流

建议条件控制流统一生成一个 `if-else` 节点作为分支门。生成步骤如下：

1. 解析 `condition_description` 得到结构化条件；
2. 创建 `if-else` 节点；
3. 对 `true`、`false`、可能的其他 case 分支分别编译子图；
4. 用带 branch 语义的边连接 `if-else` 到各分支入口；
5. 所有分支出口再通过汇合策略汇合。

对于当前骨架模型只有 `true` 分支的情况，建议强制补出 `false` 空分支，并在生成报告中标记“else 分支由系统自动补全”。这样既保持平台结构完整，也不改变原有业务语义。

### 10.4 循环控制流

循环控制流建议严格按照 Dify `iteration` 规范落地，不要自造“loop 节点”。具体步骤如下：

1. 从 `loop_condition` 与上游变量中识别迭代输入；
2. 生成 `iteration` 容器节点；
3. 生成 `iteration-start` 子节点；
4. 将循环体编译为 iteration 内部子图，所有内部节点标记 `isInIteration: true`；
5. 确定循环体的最终输出字段，绑定到 `iteration.output_selector`；
6. 将 iteration 的输出传给循环后的下游节点。

如果循环语义中包含“批量并行处理”，则 `iteration.is_parallel` 可置为 `true`，并根据任务规模或默认策略设置 `parallel_nums`。若前期难以稳定估算并发度，建议先使用固定保守值，如 3 或 5，并写成可配置项。

## 十一、参数生成策略

只有控制流是不够的，真正让 DSL 在平台上运行的是节点参数。第三阶段应当设计“参数生成优先级”。

### 11.1 参数来源优先级

建议按以下优先级决定节点参数：

1. **显式结构来源**：UTR 中已存在的 inputs、outputs、resources、variables；
2. **规则推导**：由动作名称和节点类型推导出的默认字段；
3. **平台默认模板**：预制的 Dify 节点模板；
4. **LLM 补全**：在规则不足时补生成受约束 JSON；
5. **保守默认值**：无法可靠生成时，使用安全的默认模板并打标。

### 11.2 不同节点的参数策略

- `llm`：默认模型配置走全局 profile；prompt_template 采用“系统指令 + 用户变量拼装”的最小模板；
- `code`：输入变量明确写入 `variables`；输出字段统一约定；
- `http-request`：URL、method 缺失时不应强行生成伪地址，应返回“需补充外部接口信息”的校验提示；
- `template-transform`：优先把上游变量拼成模板，不调用模型；
- `tool`：若 provider 信息不完整，不直接落地，避免生成不可用插件配置。

这意味着第三阶段既要追求自动化，也要保留诚实失败能力。不能为了“看起来生成成功”而拼出无法运行的 DSL。

## 十二、验证体系规划

第三阶段一定要配套验证体系，否则很容易出现“能生成文件但平台不能跑”的假成功。建议把验证分成四层。

### 12.1 Schema 验证

使用 Pydantic 校验内部 DSL 模型字段完整性和类型正确性。

### 12.2 图结构验证

检查节点唯一性、边合法性、入口出口完整性、循环容器结构、分支闭合、无悬挂节点等。

### 12.3 Dify 兼容验证

根据 [dify-dsl-spec-codex.md](file:///e:/Desktop/论文/工作流/utr/docs/dify-dsl-spec-codex.md) 中 Workflow 相关规范逐项核对顶层字段、节点字段、迭代结构、变量选择器格式和支持节点白名单。

### 12.4 运行前验证

如果后续条件允许，可增加一层“平台模拟验证”或“导入前校验脚本”，对 YAML 做更接近 Dify 行为的静态检查。即便短期内无法直接接入 Dify API，也应至少保证本地静态规则足够严密。

## 十五、风险与对策

### 15.1 前两阶段信息不够细

这是最大风险。对策不是回退架构，而是在阶段三引入“受约束语义补全”，同时逐步给前两阶段增加可选提示字段。

### 15.2 Dify 节点参数复杂

某些节点如 `tool`、`knowledge-retrieval` 参数复杂且依赖外部平台状态。对策是分批支持，先做核心闭环，复杂节点采用延后策略。

### 15.3 并行与汇合语义不稳定

不同平台对多前驱节点的等待行为可能存在差异。对策是默认插入显式 barrier/aggregator，牺牲一点简洁性换取稳定性。

### 15.4 LLM 补全带来不确定性

对策是严格限制模型职责、输出格式和可选值范围，所有结果必须经过程序验证后才能进入 DSL。

### 15.5 生成成功但平台运行失败

对策是建立多层验证，并尽早准备小规模真实导入测试，不把“本地序列化成功”误判为“平台兼容成功”。

## 十六、推荐的开发顺序与决策结论

综合现有项目状态与 Dify Workflow 规范，第三阶段最合理的推进方式不是直接写一个“大而全”的 `dsl_generator.py`，而是先建立可验证的编译骨架，再逐步增加节点能力。具体建议如下：

第一，先完成第三阶段最核心的基础设施：DSL Schema、图构建器、基础校验器、YAML 序列化器。没有这些，后面所有功能都会变成堆逻辑。

第二，坚持单目标推进，只围绕 Dify Workflow 模式实现，不为其他运行模式预留额外设计负担。因为你的最终目标是把生成结果放到 Dify Workflow 上运行，先聚焦最短闭环才是正确策略。

第三，先以 `start / code / llm / end / if-else / iteration / template-transform / variable-aggregator` 作为首批节点集合。这一组已经足够支撑大量业务流程，不必一开始被 `tool` 和插件依赖拖慢节奏。

第四，必须引入两层 IR，并把控制流翻译、节点映射、变量绑定、图渲染、平台校验分模块实现。否则随着并行、条件和循环叠加，代码会迅速失控。

第五，第三阶段要内建“诚实失败”机制。当某个动作缺乏足够信息时，可以生成降级节点或返回明确校验报告，但绝不能伪造平台参数。只有这样，系统才具备长期可信性。

第六，建议在阶段三开发过程中同步建立一套“典型任务样例集”，专门覆盖顺序、并行、条件、循环、文件处理和 API 调用等场景。因为第三阶段一旦没有稳定测试集，就很容易出现“这次能跑，下次又坏”的问题。

最后，从项目演进角度看，第三阶段不是一个孤立模块，而是整个系统迈向“可执行工作流生成器”的关键转折点。只要第三阶段按本报告所述方式推进，你的系统将形成一个清晰闭环：第一阶段负责把自然语言压缩成纯元数据，第二阶段负责把元数据提升为可控骨架，第三阶段负责把骨架编译成 Dify 可执行 DSL。这个闭环既符合你当前项目的研究逻辑，也符合工程落地需要，后续无论是扩展平台支持、增加评估体系，还是撰写论文方法章节，都会更加顺畅。
